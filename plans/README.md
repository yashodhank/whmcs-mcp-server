## Context the executors share (none have seen the audit)

The subsystem keeps a WHMCS API IP allowlist (`APIAllowedIPs`) current as the
operator's public IP changes. A Python CLI (`whmcs_ip_updater.py`) detects the
current IP and SSHes (restricted forced-command user) to the WHMCS host; a PHP
worker (`remote/whmcs_api_ip_updater.php`) compare-and-swaps the `MacIPv4`/
`MacIPv6` entries in the serialized `APIAllowedIPs` config row. A launchd agent
runs it every 5 min; the MCP server also runs it reactively on a `403 Invalid
IP`. The forced-command wrapper (`remote/whmcs-api-ip-updater`) strictly
allowlists actions and enforces a base64url-only payload — **do not weaken it**.

## Findings considered and rejected (do not re-audit)

- SQL injection in the worker: not present — every query uses `prepare()` +
  `bind_param()`. (`whmcs_api_ip_updater.php:100,196,212,283,432`)
- `unserialize` object injection: guarded with `['allowed_classes' => false]`
  (`whmcs_api_ip_updater.php:119`).
- FD leak in `ExecutionLock.acquire`: each invocation is a short-lived process
  that exits (launchd/MCP spawn), so the OS reclaims the fd; not a real leak.
- Provider-consensus early-exit "race": `best_ip` sorting makes extra results
  harmless.
- Missing `break` in the worker's `switch`: every `action_*` is a `never`-return
  that calls `respond()`/`exit()`, so fallthrough cannot occur.
- Unbounded state-file growth: the file is ~200 bytes with fixed keys.
